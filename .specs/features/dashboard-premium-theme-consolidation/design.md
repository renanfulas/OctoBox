# Design

## Consolidation model

The pilot output must be sorted into three boxes.

## Box 1. Export

These are candidates to become product grammar:

1. deeper navy temperature for premium dark shells
2. layered support surfaces instead of flat dark blocks
3. disciplined shell-to-topbar integration
4. richer but quieter surface borders and inset highlights
5. controlled neon that follows severity rather than decoration

## Box 2. Adapt

These may travel, but not literally:

1. hero richness
2. support rail depth
3. advisory side-card polish
4. premium card background layering

Reason:

The dashboard is a showcase control room.
Finance and reception are more transactional and may need quieter variants.

## Box 3. Keep local

These should likely stay dashboard-specific:

1. the exact intensity of glance alert glow
2. the exact hero orchestration of the dashboard shell
3. the specific balance between decision strip and side rail

## Guardrail

Export principles before exporting values.

That means:

1. export "what this effect does"
2. only then decide whether the exact token or numeric value should travel

## Pilot-to-grammar question

The consolidation question is not:

"What looked good on the dashboard?"

It is:

"What improved product perception without weakening operational truth?"

## T1 inventory result

The pilot naturally separated into three visual families.

### Family A. Shell grammar

This includes:

1. navy temperature
2. layered surface depth
3. scoped topbar integration
4. premium but quiet borders and shadows

Interpretation:

This is the strongest candidate for exportable grammar.

### Family B. Support-surface polish

This includes:

1. richer side-card depth
2. calmer premium rail composition
3. support-card refinement that does not try to dominate

Interpretation:

This should likely travel only through adapted variants, not exact cloning.

### Family C. Dashboard-local alert expressiveness

This includes:

1. exact glance-card glow intensity
2. alert pulse behavior
3. all-clear harmony state
4. the dashboard-specific emotional balance between urgency strip and side rail

Interpretation:

This is where reuse risk is highest.
The semantics can travel.
The exact visual performance probably should not.

## T2 classification decision

The pilot outputs now have an explicit destination.

### Export

These should become reusable product grammar:

1. deeper premium navy temperature for dark control-room shells
2. layered surface construction instead of flat dark panels
3. shell-scoped topbar integration through variables
4. quieter premium borders, inset light, and shadow discipline
5. the rule that neon follows severity and actionability, never decoration

Why:

These improve product coherence without depending on dashboard-specific storytelling.

### Adapt

These can travel only through calmer variants:

1. support-rail depth from `side.css`
2. richer premium card layering from glance and side surfaces
3. hero richness and shell drama when a page has true command authority
4. support-card polish for secondary rails

Why:

These are valuable, but their exact dashboard intensity may overload finance, reception, or form-heavy flows.

### Keep local

These should remain dashboard-specific unless proven otherwise:

1. exact glance glow amplitude
2. alert pulse behavior timing and strength
3. all-clear harmony treatment
4. the exact emotional balance between dashboard topline and side rail

Why:

These are part of the dashboard's stagecraft.
If copied blindly, they risk turning the rest of the product into scenery instead of operation.

## Practical rule

If a visual choice helps the product feel premium across contexts, export it.

If it only feels right because the dashboard is a showcase command room, adapt it or keep it local.

## T3 first expansion decision

### Chosen surface

The first non-dashboard surface for safe adoption should be:

1. finance command and support surfaces

Primary anchors:

1. `templates/catalog/finance.html`
2. `static/css/catalog/finance/_boards.css`

### Why finance wins

1. finance already behaves like a decision center, not a fast service desk
2. it can benefit from richer premium shell grammar without sacrificing transactional clarity
3. it already contains board-like surfaces, guide cards, and priority framing that map well to the pilot grammar

### Why reception does not win first

Anchors:

1. `templates/operations/reception.html`

Reason:

Reception is a short-ritual counter workspace.
Its visual grammar must stay lighter and quicker.
It should inherit quieter variants later, not the first expansion pass.

### Why student form does not win first

Anchors:

1. `templates/catalog/student-form.html`
2. `static/css/catalog/shared/student-financial.css`

Reason:

The student form already mixes identity, form flow, and finance.
It is a more complex surface and therefore a worse first target for premium grammar expansion.

### Expansion rule

Phase the premium grammar like this:

1. dashboard proves the language
2. finance adopts the exportable shell grammar and adapted support depth
3. reception receives quieter derivatives later
4. student form only after the grammar is stable across clearer command surfaces

## T4 governance rules

Future premium expansion must obey these guardrails.

### Rule 1. Export grammar, not dashboard theater

Allowed to travel:

1. shell temperature
2. surface layering
3. border/shadow discipline
4. semantic neon logic

Not allowed to travel by default:

1. exact glance glow values
2. dashboard all-clear choreography
3. dashboard-specific alert pulse intensity

### Rule 2. Transactional surfaces inherit quieter variants

Reception, forms, and fast operational flows must receive:

1. calmer background layering
2. lower glow amplitude
3. less atmospheric drama than dashboard and finance

### Rule 3. Command authority earns richer atmosphere

A surface may receive stronger premium treatment only when it truly acts like:

1. a command room
2. a decision center
3. a high-level operational overview

Otherwise, use quieter derivatives.

### Rule 4. Semantics stay upstream

Visual intensity must continue to follow:

1. severity
2. actionability
3. reading priority

Never reverse that relationship by styling a card as urgent before the data says it is urgent.

### Rule 5. Every expansion starts with one pilot

Do not spread new premium grammar across many surfaces in one pass.

Sequence:

1. pick one surface
2. validate clarity under pressure
3. classify what worked
4. only then expand again

## Final governance test

Before expanding any premium treatment, ask:

1. does this increase product coherence?
2. does this preserve operational truth?
3. does this help reading, not just admiration?

If any answer is no, the effect should be reduced, adapted, or kept local.
