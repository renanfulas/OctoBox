# Trilha de leitura para Dev Junior

## Objetivo

Esta trilha existe para ajudar uma pessoa junior a:

1. entender o produto
2. entender a organizacao do projeto
3. saber por onde ler sem se perder
4. aprender termos tecnicos sem pular direto para a parte mais pesada

## Ordem recomendada

1. [../../../README.md](../../../README.md)
2. [../general-architecture-guide.md](../general-architecture-guide.md)
3. [../methodology-and-organization-guide.md](../methodology-and-organization-guide.md)
4. [../../reference/documentation-authority-map.md](../../reference/documentation-authority-map.md)
5. [../../reference/reading-guide.md](../../reference/reading-guide.md)

## O que a pessoa precisa sair sabendo

1. o que o OctoBox resolve
2. que `boxcore` nao e mais a melhor explicacao do runtime novo
3. que a base separa leitura, regra e apresentacao
4. que docs tem pesos diferentes

## Depois disso

Escolha uma area:

1. se for frontend, siga para [frontend-reading-path.md](./frontend-reading-path.md)
2. se for backend, siga para [backend-reading-path.md](./backend-reading-path.md)
3. se for deploy, siga para [deploy-reading-path.md](./deploy-reading-path.md)

## O que nao fazer cedo demais

1. sair abrindo `boxcore` como se fosse o centro de tudo
2. ler plano profundo antes de entender o mapa geral
3. entrar em CSS local sem entender a camada global

## Metafora simples

Primeiro voce aprende o mapa do shopping.

So depois voce vai para a loja certa.

Se entrar pela porta do estoque sem conhecer o corredor, tudo parece caos mesmo quando nao e.
