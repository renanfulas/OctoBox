## Achados estruturais (alem dos numeros)

### Achado 1: testes order-dependentes ja existem
`pytest-randomly` esta em `requirements_test.txt`. No run aleatorio
(seed gerado automaticamente), padroes de falha sugerem que alguns
testes dependem de ordem (ex.: blocos contiguos de F que somem em
ordem fixa). Quantificar exige Fase 3, mas o sintoma esta presente.

### Achado 2: 2 arquivos tem DB access em import-time
`test_csv_injection.py` e `test_whale.py` falham na fase de
**coleta** (antes de qualquer fixture rodar). Causa: `Model.objects`
queries em module level. Fix: mover para fixtures ou para `setUp`.

### Achado 3: muitas falhas sao pre-existentes, NAO causadas pelo conftest
Verificado rodando 3 arquivos individuais — failures aparecem
mesmo com conftest funcionando perfeitamente para os 14 testes
do CI atual. Conftest auto-Membership e schema_context estao OK;
o problema sao tests que dependem de premissas que mudaram
(provavelmente Sprint 2 schema-per-tenant ou refactors antigos).

### Achado 4: cobertura real desconhecida ate suite rodar inteira
Os 14 testes do CI atual cobrem 8.3%. Os 882 restantes provavelmente
levam isso para 50-60% (estimativa baseada em "892 testes nao podem
estar todos cobrindo as mesmas 8.3%"). Numero exato sera revelado
em Fase 1 apos Fase 0.5.

## Decisao: replanejar para inserir Fase 0.5

### Fase 0.5 — Suite Stabilization (NOVA, S-M, 1-3 dias)

**Objetivo:** Tornar a suite verde em ordem fixa antes de:
- Adicionar ao CI (Fase 1 — sem isso, CI fica vermelho cronicamente)
- Medir cobertura honesta (Fase 0 reload — o numero atual e inutil)
- Detectar order-dependence (Fase 3 — testes ja quebrados em qualquer
  ordem mascaram order-dependence)

**Passos:**
1. Rodar suite em ordem fixa (`-p no:randomly`) num arquivo por vez,
   coletando lista de FALHAS pre-existentes.
2. Para cada falha: triagem em 3 buckets
   - **Trivial** (1-2h): test code stale (ex.: usa API antiga, expects
     valor que mudou). Fix imediato.
   - **Sprint 2 cross-schema** (M): test depende de FK removida ou
     comportamento mudado pela migration schema-per-tenant. Fix usa
     factories / box_test schema.
   - **Bug real do produto descoberto pelo teste** (raro): abrir
     issue + decidir fix vs. skip-com-bug-ID.
3. Corrigir bucket 1 e 2 em ordem de impacto (apps mais usados primeiro:
   boxcore, finance, operations, catalog).
4. Confirmar: `python -m pytest --reuse-db -p no:randomly` passa 100%.
5. **Daí prosseguir para Fase 1** (CI roda full suite).

**Trade-off:** Pode levar 1-3 dias dependendo de quantos testes
estao quebrados. Sem isso, Fases 1-9 nao geram valor real.

**Fallback se Fase 0.5 for inviavel agora:** Marker `@pytest.mark.broken`
em todos os testes quebrados + adicionar `-m "not broken"` no CI.
Lista o que esta marcado em `docs/testing/broken-tests-inventory.md`.
Isso e dívida explícita, nao limpeza, mas desbloqueia Fase 1.

import psycopg, os
url = os.environ['DATABASE_URL'].replace('octobox_control', 'postgres')

    with c.cursor() as cur:
        cur.execute(\"SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE datname='test_octobox_control' AND pid != pg_backend_pid()\")
